배포 환경 변경 공유 문서

팀원 공유용: 로컬 개발, 서버 배포, GitHub Actions, RDS, S3, 더미 데이터 운영 방식 변경 정리

한 줄 요약

로컬 개발은 local 프로필에서 자동 더미 생성, 서버 배포는 cloud 프로필에서 GitHub Secrets/Variables와 RDS/S3를 사용하는 방식으로 분리했습니다.

왜 바꿨나

항목 기존 방식 변경 방식 바꾼 이유 장점 관리 방법
실행 환경로컬과 서버가 같은 설정을 많이 공유local, cloud 프로필 분리로컬 설정이 서버에 섞여서 배포 오류가 발생했음로컬 작업과 서버 배포가 서로 덜 충돌함로컬은 application-local.properties, 서버는 application-cloud.properties 기준
기본 프로필서버/로컬 구분이 애매함기본값은 local, 배포 때만 SPRING_PROFILES_ACTIVE=cloud개발자는 그냥 실행하고 서버는 운영 설정을 명시적으로 쓰게 하기 위함로컬 실행이 쉬워지고 배포 설정 실수가 줄어듦서버 Docker env에는 반드시 SPRING_PROFILES_ACTIVE=cloud가 들어감
DB 설정코드 안의 고정값 또는 수동 주석 전환DB_URL, DB_USERNAME, DB_PASSWORD env 사용로컬 DB와 RDS 주소가 다르기 때문RDS 주소/계정이 코드에 박히지 않음GitHub Actions Secrets 또는 Variables에 등록
DDL 전략서버에서도 create가 섞일 위험로컬은 create, 서버는 기본 validate서버 재배포 때 테이블/데이터가 날아가는 위험 차단배포해도 RDS 데이터가 유지됨서버 테이블 변경은 migration/DDL을 먼저 반영하고 배포
더미 데이터서버 시작 시 코드가 자동 삽입할 수 있음로컬만 DataInitializer 실행, 서버는 SQL DML 직접 실행서버 재시작마다 더미가 중복/초기화되는 문제 방지서버 데이터 주입 시점을 사람이 통제 가능테스트 서버 더미는 workation/back/dev-seed/test-open-dummy-data.sql을 pgAdmin에서 실행
더미 실행 조건DataInitializer가 항상 돌 수 있음app.seed.enabled=true일 때만 실행서버에서는 더미 자동 실행을 막기 위함로컬/서버 데이터 흐름이 명확해짐로컬 기본값은 true, cloud 기본값은 false
서버 더미 삽입앱 재시작에 의존pgAdmin/psql로 DML 실행배포와 데이터 삽입을 분리하기 위함배포 실패와 데이터 실패를 따로 확인 가능기존 데이터 삭제 후 넣을 때는 TRUNCATE ... RESTART IDENTITY CASCADE 먼저 실행
더미 이미지jar 내부 정적 파일에 포함될 수 있음S3에 업로드하고 DB에는 S3 key 저장jar 용량 증가와 서버 정적 파일 의존을 줄이기 위함이미지가 서버 컨테이너와 분리되어 안정적임이미지 원본은 repo에 있고 서버 노출은 S3 URL로 함
jar 이미지 포함static/dummy-images가 jar에 들어갈 수 있음bootJar에서 static/dummy-images/** 제외이미지 때문에 백엔드 jar가 커지는 문제 방지빌드/배포가 가벼워짐이미지가 필요하면 S3 업로드 스크립트 사용
S3 업로드수동 업로드만 가능upload-dummy-images-to-s3.ps1 추가더미 이미지 대량 업로드를 반복 가능하게 하기 위함같은 경로로 재업로드하기 쉬움workation/back/dev-seed/upload-dummy-images-to-s3.ps1 사용
S3 주소화면마다 URL 조합 방식이 제각각VITE_S3_ASSET_BASE_URL, resolveS3AssetUrl 사용로컬/서버에서 이미지 경로가 다르게 깨지는 문제 방지S3 base URL만 바꾸면 전체 화면이 따라감프론트 env는 workation/front/.env.example 참고
S3 자격증명access key가 코드에 들어갈 위험GitHub Secrets 또는 AWS 기본 자격증명 사용키 노출 방지보안 관리가 쉬워짐AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY는 GitHub Secrets에만 저장
S3Configaccess key/secret이 없으면 실패 가능없으면 DefaultCredentialsProvider, 있으면 static credentials서버 IAM Role 또는 GitHub Secret 둘 다 대응하기 위함배포 환경 선택지가 넓어짐운영에서는 가능한 IAM Role 또는 Secrets 중 하나로 일관되게 관리
프론트 .env.env, .env.production이 repo에 들어갈 수 있음실제 env 파일은 제외하고 .env.example만 유지개인/운영 설정 충돌 방지팀원별 로컬 설정 충돌 감소각자 .env는 로컬에서 만들고 커밋하지 않음
프론트 API 주소하드코딩 URL이 섞임VITE_API_BASE_URL, VITE_SERVER_BASE_URL, VITE_WS_URL 사용로컬과 배포 API 주소가 다르기 때문API 주소 변경 시 env만 바꾸면 됨GitHub Variables 또는 workflow 기본값으로 관리
axios baseURL코드 안에서 직접 관리src/app/config/env.js에서 env 기반 관리여러 파일에서 API 주소가 달라지는 문제 방지주소 관리 지점이 하나로 모임API 주소를 바꿀 때는 env.js와 GitHub Variables 확인
WebSocket 주소로컬/서버 주소가 섞일 수 있음VITE_WS_URL 사용배포 도메인은 wss, 로컬은 ws가 필요함HTTPS 배포에서 WebSocket 주소가 명확해짐서버는 wss://api.sandykey.shop/ws-connect 기준
CORS로컬 origin 중심cloud는 https://sandykey.shop 허용배포 프론트 도메인에서 API 호출해야 하기 때문배포 브라우저 요청이 막히지 않음추가 도메인은 APP_CORS_ALLOWED_ORIGINS로 관리
백엔드 포트로컬/서버 포트 혼동 가능앱 내부 기본 8001, Docker candidate는 8002:8001후보 컨테이너 health check를 하기 위함새 컨테이너 확인 후 기존 컨테이너 교체 가능서버 외부 매핑은 workflow의 Docker run 명령 확인
backend deploy빌드와 배포가 섞여 원인 파악 어려움Docker build/push 후 EC2에서 candidate health check배포 실패 시 기존 컨테이너를 바로 죽이지 않기 위함실패해도 기존 서비스 유지 가능실패 로그는 docker logs sandykeyboard-candidate 확인
deploy 사전 검증누락 Secret을 health check 단계에서야 알 수 있음Validate deployment configuration 단계 추가DB_URL 누락 같은 실수를 빠르게 찾기 위함어떤 설정이 빠졌는지 Actions에서 바로 보임필수 값이 없으면 Docker 실행 전에 실패
메일 health check메일 비밀번호 없으면 health check 실패cloud에서 management.health.mail.enabled=false 기본값메일 계정이 없어도 서버 기동 확인은 가능해야 했음배포가 메일 설정 때문에 막히지 않음메일 상태까지 검사하려면 MAIL_HEALTH_ENABLED=true와 메일 계정 등록
CIPR 검증과 배포가 섞일 수 있음ci.yml은 PR 빌드 검증만 담당main 배포 전에 빌드 오류를 먼저 잡기 위함PR 단계에서 백/프론트 빌드 실패 확인 가능PR 생성 시 Backend, Frontend CI 통과 확인
CDCI와 배포 구분이 약함backend-deploy.yml, frontend-deploy.yml 분리백엔드/프론트 변경 범위가 다르기 때문필요한 쪽만 배포됨main push 시 변경 path에 따라 배포 workflow 실행
backend CDmain push 시 백엔드 관련 파일만 배포.github/workflows/backend-deploy.yml, workation/back/** 변경 시 실행프론트만 바꿨는데 백엔드까지 배포되는 낭비 방지배포 시간/위험 감소백엔드 변경 PR merge 후 Backend Deploy 확인
frontend CDmain push 시 프론트 관련 파일만 배포.github/workflows/frontend-deploy.yml, workation/front/** 변경 시 실행백엔드만 바꿨는데 프론트까지 배포되는 낭비 방지배포 범위가 명확함프론트 변경 PR merge 후 Frontend Deploy 확인
Docker 이미지latest만 쓰면 롤백 추적이 어려움SHA 태그와 latest 둘 다 push어떤 커밋 이미지인지 추적하기 위함문제 발생 시 커밋 단위 확인 가능Actions 로그의 이미지 태그가 커밋 SHA
Actuatorhealth endpoint가 필요spring-boot-starter-actuator 추가배포 후보 컨테이너가 정상인지 자동 확인하기 위함사람이 직접 접속 확인하기 전에 배포가 검증됨/actuator/health가 200이어야 컨테이너 교체
GitHub 설정 저장 위치APPLICATION_PROPERTIES 같은 통짜 변수 사용 가능성개별 Secrets/Variables 사용한 파일 문자열로 관리하면 충돌/누락/노출 위험이 큼무엇이 필요한지 이름으로 바로 확인 가능통짜 APPLICATION_PROPERTIES는 사용하지 않음

로컬 개발 방법

할 일한 줄 설명
백엔드 실행workation/back에서 ./gradlew bootRun 실행
프론트 실행workation/front에서 npm install 후 npm run dev 실행
로컬 DB기본값은 jdbc:postgresql://localhost:5432/code_hit
로컬 DDL기본값은 create라서 실행 시 테이블을 다시 만들 수 있음
로컬 더미기본값은 app.seed.enabled=true라서 DataInitializer가 더미를 넣음
로컬 설정 변경LOCAL_DB_URL, LOCAL_DB_USERNAME, LOCAL_DB_PASSWORD 같은 env로 덮어씀
로컬 프론트 env.env.example을 참고해서 각자 .env를 만들고 커밋하지 않음

서버 배포 방법

할 일한 줄 설명
main에 mergemain push가 배포 트리거임
백엔드 배포backend workflow가 jar build, Docker build/push, EC2 deploy, health check 수행
프론트 배포frontend workflow가 Vite build, S3 sync, CloudFront invalidation 수행
서버 프로필Docker env로 SPRING_PROFILES_ACTIVE=cloud가 들어감
서버 DBDB_URL, DB_USERNAME, DB_PASSWORD로 RDS에 접속함
서버 DDL기본 validate라서 테이블이 없으면 앱이 실패함
서버 더미앱이 자동으로 넣지 않고 pgAdmin/psql로 DML을 직접 실행함
배포 실패 확인GitHub Actions의 실패 step을 펼쳐서 에러 원문을 확인함

pgAdmin으로 더미 넣는 순서

  1. RDS DB에 접속한다.
  2. Query Tool을 연다.
  3. 테스트 DB를 비워도 되는 상황이면 아래 초기화 SQL을 먼저 실행한다.
  4. test-open-dummy-data.sql 전체를 붙여넣고 실행한다.
  5. 실행 후 user0001 / user1234!로 로그인 확인한다.
DO $$
DECLARE
    r RECORD;
BEGIN
    FOR r IN (
        SELECT tablename
        FROM pg_tables
        WHERE schemaname = 'public'
    ) LOOP
        EXECUTE format('TRUNCATE TABLE public.%I RESTART IDENTITY CASCADE;', r.tablename);
    END LOOP;
END $$;

팀원이 기억할 규칙

규칙한 줄 설명
Secret 값은 코드에 쓰지 않기비밀번호, 토큰, access key는 GitHub Secrets에만 둠
.env는 커밋하지 않기개인 로컬 설정이 다른 팀원/서버와 충돌할 수 있음
서버는 cloud만 사용서버에서 local 프로필이 돌면 DDL create와 더미 자동 삽입 위험이 있음
서버 DB는 자동 초기화하지 않기운영/테스트 데이터가 날아갈 수 있음
더미는 직접 DML로 넣기배포와 데이터 삽입을 분리해야 문제 원인 파악이 쉬움
이미지 파일은 S3 기준으로 보기서버 jar 안의 정적 파일에 의존하지 않음
Actions 실패는 실패 step을 펼쳐보기맨 위 에러보다 아래 실제 Spring/Docker 로그가 원인임
main push는 배포임main에 merge하면 자동으로 배포가 돈다고 생각해야 함
PR에서는 CI만 확인PR은 build 검증, main merge 후 CD 배포 확인
설정 추가 시 문서도 업데이트새 env가 생기면 이 문서와 .env.example에 같이 적음
